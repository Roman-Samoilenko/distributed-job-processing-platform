# Регламент разработки (Workflow)

Этот документ описывает правила ведения задач, именования веток и коммитов для проекта **Distributed Job Processing Platform**.

## 1. GitHub Issues (Задачи)

### Когда создавать Issue?
*   **ОБЯЗАТЕЛЬНО:**
    *   Новая функциональность (Feature).
    *   Сложный баг, требующий расследования.
    *   Архитектурные изменения.
*   **НЕ ОБЯЗАТЕЛЬНО (можно сразу делать ветку):**
    *   Мелкие правки документации (опечатки).
    *   Стилевые правки (formatting).
    *   Мелкие конфигурационные изменения.
    *   Задачи, занимающие менее 15 минут ("Правило 15 минут").

### Правила оформления Issue:
1.  **Заголовок:** На русском языке, кратко и емко.
2.  **Описание:** Что конкретно нужно сделать, критерии приемки.
3.  **Labels (Метки):** Обязательно проставить тип задачи.
4.  **Assignees:** Назначить себя исполнителем.

---

## 2. Ветки (Branches)

Работать в ветке `main` **запрещено**. Для каждой задачи (даже мелкой) создается отдельная ветка.

### Правила именования:
Используем **английский язык** и kebab-case.

**Формат:**
`тип/короткое-описание-на-английском`

### Список типов:
*   `feat` — новая функциональность.
*   `fix` — исправление ошибки.
*   `chore` — настройка, рутина, обновление зависимостей.
*   `refactor` — рефакторинг без изменения логики.
*   `docs` — документация.
*   `test` — тесты.

### Примеры:
*   `feat/add-config-loader` (крупная задача под Issue)
*   `fix/readme-typo` (мелкая правка без Issue)

---

## 3. Коммиты (Commits)

Делаем коммиты часто. Один логический шаг = один коммит.

### Формат:
`тип: описание на русском`

**Важно:**
1.  **Тип** пишем на английском (строго из списка ниже).
2.  После двоеточия — **пробел**.
3.  **Описание** пишем на русском языке, с маленькой буквы.
4.  (Опционально) В конце можно добавить `(Fixes #номер_issue)`, если задача привязана к Issue.

### Примеры:
*   `feat: добавил чтение KAFKA_BROKERS из env (Fixes #5)`
*   `docs: исправил опечатку в инструкции запуска`

---

## 4. Алгоритм работы

### Вариант А: Большая задача (Feature / Bug)
1.  Создай **Issue** ("Реализовать Worker Pool", #10).
2.  Создай ветку: `git checkout -b feat/worker-pool`.
3.  Напиши код.
4.  Коммит: `feat: реализовал базовый пул воркеров (Fixes #10)`.
5.  Отправь PR и смержи.

### Вариант Б: Мелкая правка (Fast Track)
1.  Issue создавать **не нужно**.
2.  Создай ветку: `git checkout -b fix/typo-config`.
3.  Исправь код.
4.  Коммит: `fix: исправил имя переменной в конфиге`.
5.  Отправь PR и смержи.
